Generative DDD

Generative DDD

A new methodology for solo entrepreneurs to build software faster by focusing on domain understanding with AI assistance.

This book explores how AI reshapes Domain-Driven Design — reducing the need for traditional domain experts while amplifying individual developer power.

Introduction

What is Generative DDD? This chapter explores the vision and goals of our methodology.

A Running Example: Ethan’s AI-Powered GTD App

To ground these abstract Domain-Driven Design concepts into something practical, we will follow a single, evolving example throughout this book.

Meet Ethan. Ethan is a solo entrepreneur and developer. Like many of us, he suffers from digital information overload. He has ideas scattered across GitHub issues, Telegram messages, emails, and dozens of open browser tabs.

Ethan decides to build an application inspired by the Getting Things Done (GTD) methodology. However, unlike standard TODO lists (which require manual entry), Ethan’s vision is a Zero-Friction AI-GTD App.

The defining feature—the “Core Domain” where the business magic happens—is an AI Auto-Capture Engine. This engine runs in the background, continuously analyzing Ethan’s digital footprint (reading Telegram chats, monitoring GitHub assignments, scanning emails) and using a Large Language Model to automatically extract actionable tasks and route them into his GTD Inbox.

Why this Example?

This hypothetical application is perfect for learning DDD because it possesses real domain complexity:

  1. Core Domain: The AI Auto-Capture Engine (Intelligent parsing, deduplication, and routing).
  2. Supporting Domains: Integrations and Connectors (The infrastructure communicating with the GitHub API, Telegram bots, and Gmail).
  3. Generic Domains: Standard Task Management (The lists themselves: Next Actions, Waiting For, Projects, Someday/Maybe).

Throughout Chapter 2, we will watch Ethan attempt to model this complex system using the Traditional DDD Starter Process—and experience the friction of doing it alone. Then, in the subsequent chapters, we will observe how Ethan accelerates this same process using Generative DDD.

Process Overview: The Starter Modelling Process

In this chapter, we audit the DDD Starter Modelling Process, a step-by-step guide designed to help teams navigate the complexity of Domain-Driven Design.

Credit: The content of this chapter, including the 8-step process and descriptions, is adapted from the Domain-Driven Design Starter Modelling Process by the DDD Crew, which is licensed under CC-BY-SA-4.0.

Overview of the Process

The process is composed of 8 iterative steps, grouped into four phases of sociotechnical architecture:

Phase 1: Align & Understand

  1. Understand: Align focus with the business model and user needs.
  2. Discover: Collaborative discovery of the domain.

Phase 2: Strategic Architecture

  1. Decompose: Breaking the domain into sub-domains.
  2. Strategize: Identifying core domains for business differentiation.

Phase 3: Strategy & Org Design

  1. Connect: Designing interactions between sub-domains to minimize coupling.
  2. Organise: Aligning autonomous teams with context boundaries.

Phase 4: Tactical Architecture

  1. Define: Explicitly defining roles of each bounded context.
  2. Code: Implementing the domain model.

Note

For solo entrepreneurs, some of these steps (like Team Topologies) might seem overkill. In later chapters, we will explore how to “generate” these steps or simplify them.

Phase 1: Align & Understand

This phase focuses on aligning the project with business goals and discovering the problem domain through collaborative techniques.

Steps

  1. Understand: Align focus with the business model and user needs.
  2. Discover: Collaborative discovery of the domain.

Step 1: Understand

Key Question: What is the business goal?

Explanation

Align our focus with the organisation’s business model, the needs of its users, and its short, medium, and long-term goals. Every decision we take regarding the architecture, the code, or the organisation has business and user consequences. In order to design, build, and evolve software systems most effectively, our decisions need to create the optimal business impact, which can only be achieved if we are aligned to the business goals, as well as supporting the users current and potential future needs.

Recommendation

As a starting point, we recommend The Business Model Canvas for the business perspective, and User Story Mapping for understanding the user vantage point.

Tools & Techniques

  • Business Model Canvas
  • User Story Mapping
  • Impact Mapping
  • Product Strategy Canvas
  • Wardley Mapping

Step 2: Discover

Key Question: How does the process flow?

Explanation

Discover the domain visually and collaboratively. This is the most crucial aspect of DDD. You cannot skip discovery. If your whole team doesn’t build up a good understanding of the domain, all software decisions will be misguided. Spreading domain knowledge through the whole team will create a shared understanding. It enables the developers to build a software system aligned to the domain which can be more flexible to incorporate future business changes.

Recommendation

As a starting point, we recommend EventStorming.

Tools & Techniques

  • EventStorming
  • Domain Storytelling
  • Example Mapping
  • User Journey Mapping
  • User Story Mapping

Phase 2: Strategic Architecture

This phase focuses on breaking the domain into manageable parts and identifying which parts are most critical for business success.

Steps

  1. Decompose: Breaking the domain into sub-domains.
  2. Strategize: Identifying core domains for business differentiation.

Step 3: Decompose

Key Question: What are the natural boundaries?

Explanation

Decompose the domain into sub-domains - loosely-coupled parts of the domain. We decompose a large problem domain into sub-domains for a few key reasons:

  • reduced cognitive load, so that we can reason about parts of the domain independently,
  • give development teams autonomy, so that they can work on separate parts of the solution,
  • identifying loose-coupling and high-cohesion in the domain which carries over to our software architecture and team structure.

Recommendation

As a starting point, we recommend carving up your event storm into sub-domains and Context Maps.

Tools & Techniques

  • Business Capability Modelling
  • EventStorming with sub-domains
  • Context Maps
  • Design Heuristics
  • Independent Service Heuristics

Step 4: Strategize

Key Question: Where is the “magic”?

Explanation

Strategically map out your sub-domains to identify core domains: the parts of the domain which have the greatest potential for business differentiation or strategic significance. Time and resources are limited, so understanding which parts of the domain to focus on is critical to delivering optimal business impact. By analysing what your core domains are, you will have a better idea of how much quality and rigour is required to build each part of your system.

Recommendation

As a starting point, we recommend Core Domain Charts.

Tools & Techniques

  • Core Domain Charts
  • Purpose Alignment Model
  • Wardley Mapping

Phase 3: Strategy & Org Design

This phase focuses on how different parts of the system interact and how the people/teams are organized around those boundaries.

Steps

  1. Connect: Designing interactions between sub-domains.
  2. Organise: Aligning autonomous focus/teams with context boundaries.

Step 5: Connect

Key Question: How do parts talk?

Explanation

Connect the sub-domains into a loosely-coupled architecture which fulfills end-to-end business use-cases. It is imperative to not only decompose a large domain into parts but to also carefully design the interactions between those parts to minimise unwanted coupling and complexity. It is necessary to challenge the initial design by applying concrete use-cases to uncover hidden complexity.

Recommendation

As a starting point, we recommend Domain Message Flow Modelling.

Tools & Techniques

  • Domain Message Flow Modelling
  • Process Modelling EventStorming
  • Sequence Diagrams
  • BPMN

Step 6: Organise

Key Question: (Solo focus) What is my focus?

Explanation

Organise autonomous teams that are optimised for fast flow and aligned with context boundaries. Teams need to be organised to have autonomy, clear goals and sense of purpose. In order to do that we need to take into account organisational constraints, so that teams organise themselves for fast flow. For solo entrepreneurs, this is mainly about organising your focus and cognitive load sequentially.

Recommendation

As a starting point, we recommend visualising sociotechnical architecture with the Context Maps.

Tools & Techniques

  • Team Topologies
  • Context Maps
  • Dynamic Reteaming
  • Explorers, Villagers & Town Planners

Phase 4: Tactical Architecture

This phase focuses on the detailed design and implementation of each bounded context.

Steps

  1. Define: Explicitly defining roles of each bounded context.
  2. Code: Implementing the domain model.

Step 7: Define

Key Question: What does this context do?

Explanation

Define the roles and responsibilities of each bounded context. Before committing to a design, make explicit decisions about the choices which can have a significant impact on the overall design. Have these conversations early while it is still easy to change your mind and explore alternative models. Design collaboratively and visually, and start to consider the technical limitations so that you can uncover constraints or opportunities.

Recommendation

As a starting point, we recommend the Bounded Context Canvas.

Tools & Techniques

  • Bounded Context Canvas
  • C4 System Context Diagram
  • Quality Storming

Step 8: Code

Key Question: Does the model fit?

Explanation

Code the domain model. Aligning the code to the domain makes it easier to change the code when the domain changes. By collaboratively modelling the problem space with experts, the developers have a chance to learn about the domain and minimise misunderstandings.

Recommendation

As a starting point, we recommend the Aggregate Design Canvas.

Tools & Techniques

  • Aggregate Design Canvas
  • Hexagonal/Onion Architecture
  • Event Modeling
  • Design-Level EventStorming
  • C4 Component Diagrams
  • Model Exploration Whirlpool

Solo Brainstorming Tips

  • Start Small: Don’t try to map the whole universe. Focus on one core flow.
  • Visual over Text: Use digital whiteboards (Miro, Excalidraw) for EventStorming.
  • The “Expert” Hat: Since you are solo, swap “hats”. Spend 30 mins as the “Business Owner”, then 30 mins as the “Developer”.
  • AI as Peer: Use AI to challenge your boundaries. Ask: “What are the hidden dependencies in this sub-domain?”
